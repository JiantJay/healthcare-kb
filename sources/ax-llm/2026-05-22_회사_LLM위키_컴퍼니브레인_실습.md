## 핵심 요약 (두괄식)

**회사의 진짜 자산은 매출 데이터가 아니라 "왜 그렇게 했는지"(결정의 맥락·판단)이며, 이를 사람 머릿속·죽은 노션 페이지에서 꺼내 AI가 직접 쓰고 관리하는 "살아 있는 회사 위키피디아(컴퍼니 브레인)"로 만드는 것이 AI 네이티브 회사로 가는 한 걸음이다.** Obsidian(유리창) + Claude Code(24시간 사서) 조합으로 코드 없이 회사용 LLM Wiki를 세팅하고, 47개 사내 문서를 자동 분류·연결한 뒤 "가격 정책 변천"·"과거 유사 고객 불만 해결법" 같은 질의로 활용하는 전 과정을 시연한다. 사람이 할 일은 "자료 던지기와 질문하기" 둘뿐이다.

주요 논점:
- **회사 기억상실증**: 결정의 맥락이 누군가의 머릿속, 3년 전 Slack, 아무도 안 보는 노션, 제목도 기억 안 나는 회의록에 흩어짐 → 퇴사와 함께 증발, 신입은 백지에서 시작, 10년 된 회사가 1년차 실수를 반복
- **LLM Wiki의 차이**: 기존 AI는 질문마다 원본 더미를 재탐색. LLM Wiki는 자료 유입 시 한 번 정리해 페이지로 "컴파일"(매번 요리가 아니라 만들어둔 반찬을 꺼내 먹기) → 빠르고 일관되며 쌓일수록 강해짐(복리)
- **구조 4요소**: ① 창고 2칸(원본 창고 raw는 불변 보관 / 백과사전 wiki) ② 페이지 1개 = 개념 1개(한 줄 요약+태그+본문) ③ 페이지 간 링크로 지식 그물망 ④ AI = 사서 겸 편집장(갱신·신규 생성·**모순 표시**·외톨이 페이지·빠진 개념 점검)
- **구축 절차(노코드)**: Obsidian 볼트 생성 → Claude Code 실행 → Karpathy 원문 링크 + "시작 전 나에게 질문하라(주제 범위·자료 종류·페이지 종류)" 프롬프트 → 답변 기반으로 CLAUDE.md 운영규칙·폴더 구조·위키 인덱스 자동 생성 → raw에 문서 47개 투입 → "CLAUDE.md 규칙대로 위키 정리" → 인물·프로세스·프로젝트·결정·고객 자동 분류, 그래프 뷰에서 백링크 기반 상관계도("작은 온톨로지") 확인
- **활용 질의 패턴**: "가격 정책이 왜·어떻게 바뀌었고 결과는?"(시간순), "대형 고객의 유사 요금 불만 — 과거 해결법·효과·주의점·반복 금지 실수 정리, **출처 표기, 위키에 없으면 '없음'이라 답하라**" → 백지가 아닌 축적된 경험 위에서 결정

> ⚠️ **수치·표현 주의**: "온보딩 3개월→3일", "복리로 똑똑해진다" 등은 발표자의 기대 효과 표현이며 검증된 수치가 아닙니다. "노션 장례식"이라는 제목도 Notion 도구 자체의 대체가 아니라 "아무도 안 보는 죽은 문서 vs AI가 유지하는 살아있는 위키"의 대비를 위한 도발적 프레이밍입니다. 시연 데이터는 발표자가 준비한 가상 SaaS 회사(30명 규모) 문서 기준입니다.

## 주제별 정리

### 1. 문제 정의 — 회사 기억상실증
- 회사의 진짜 자산 = "왜 이 고객을 잡았고, 왜 저 프로젝트를 접었고, 왜 이 가격으로 갔는지"의 맥락과 판단
- 현재 소재지: 누군가의 머릿속 / 3년 전 Slack / 아무도 안 들어가는 노션 페이지 / 제목 기억 안 나는 회의록
- 결과: 인재 퇴사 시 증발, 신입 백지 출발(온보딩 장기화), 같은 실수의 반복

### 2. 회사 단위 LLM Wiki 개념
- Karpathy의 LLM Wiki(발표자: 올해 4월 제시⚠️)를 회사 단위로 확장 — "AI가 직접 쓰고 연결하고 관리하는 살아있는 위키피디아"
- 새 자료 유입 → AI가 핵심을 페이지로 추출 → "이 결정은 작년 그 프로젝트와 연결" 자동 연결 → "이건 저번 방침과 충돌" **모순 적발**
- 기존 방식(질문마다 원본 재탐색)과 달리 **유입 시 1회 정리(컴파일)** 후 정리본만 읽음 → 빠름·일관성·복리 축적

### 3. 구조 4요소
1. **창고 2칸**: raw(회의록·문서·기록 원본, 불변 보관) / wiki(AI가 가공한 백과사전)
2. **1페이지 = 1개념**: 한 줄 요약 + 태그 + 본문 (예: "우리 회사 가격 정책", "○○ 고객사")
3. **페이지 간 링크**: 위키피디아의 파란 글씨처럼 — 가격 정책 페이지 안에 ○○고객사 협상 링크 → 흩어진 지식이 그물망화
4. **AI = 사서 겸 편집장**: 관련 페이지 갱신 / 없으면 신규 생성 / 충돌 표시 / 외톨이 페이지·누락 개념 점검

### 4. 구축 실습 (노코드)
- 역할 분담: **Obsidian = 지식 창고를 들여다보는 유리창 / Claude Code = 24시간 사서 / wiki = 집단 기억 도서관**. 사람은 "자료 던지기 + 질문하기"만
- 절차:
  1. Obsidian 볼트 생성(예: Company Brain)
  2. 터미널에서 해당 폴더로 이동 후 `claude` 실행
  3. Karpathy LLM Wiki 원문 링크 + 프롬프트: "이 파일을 읽고 이 폴더를 우리 회사용 LLM 위키로 세팅하라. **시작 전 나에게 물어라** — 위키의 주제 범위, 처음 들어올 자료 종류, 필요한 페이지 종류. 답변에 맞는 CLAUDE.md 운영규칙·기본 폴더 구조·위키 인덱스를 만들어라"
  4. 답변 예시: "30명 규모 SaaS 전사 / 의사결정 기록·고객 협상 회의록·온보딩 프로세스·분석 / 추천한 6종 그대로. **출처 표기와 모순 체크는 필수**"
  5. 생성 결과: CLAUDE.md, README, decisions·clients·processes 등 섹션 + raw/wiki 폴더 구조
  6. raw에 사내 문서 47개 투입 → "raw에 새 자료를 넣었다. CLAUDE.md 규칙대로 위키에 정리하고, 끝나면 만든 것과 연결한 것을 요약하라"
  7. 자동 분류: 인물(역할·오너십·백링크), 프로세스, 프로젝트, 소스(의사결정 시점), decisions, 고객 → 그래프 뷰에서 상관계도 확인("작은 온톨로지")

### 5. 활용 질의 패턴 (시연)
- **히스토리 복원**: "우리 가격 정책이 어떻게 바뀌었는지, 각각의 이유와 결과를 시간순으로 정리해 달라" → 신입·기억 안 나는 구성원도 즉시 맥락 확보
- **과거 경험 기반 의사결정**: "또 다른 대형 고객이 비슷한 요금 불만을 제기했다. 위키에서 과거 유사 상황의 해결법, 효과 있었던 것, 주의점, 반복하면 안 될 실수를 정리하라. **출처를 표기하고, 위키에 없으면 '없음'이라고 답하라**" → "일회성 할인으로 막지 마라" 같은 교훈 추출
- 핵심 효과: 매번 백지에서 결정하지 않고 **지난 경험 위에서 결정**

### 6. 마무리 제언
- 시작은 "오늘 회의록 하나, 결정 기록 하나"부터 — 그것이 슈퍼브레인의 첫 페이지
- 발표자가 세팅 프롬프트·회사 지식 정립 템플릿을 노션 링크로 공유(영상 설명란)
