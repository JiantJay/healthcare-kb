## 핵심 요약 (두괄식)

**LLM Wiki는 "질문할 때마다 재검색(RAG)"이 아니라 "문서가 들어올 때 한 번 정리·연결(write-time compile)"하는 지식 저장소이며, 성패는 초기 구조가 아니라 '문서 품질을 보존하는 유지보수의 자동화 + 사람의 최종 승인'에 달려 있다.** 발표자는 Andrej Karpathy의 LLM Wiki 개념을 개발자·비개발자 혼합 스타트업에서 Git 저장소 + Obsidian 조합("Knowledge Context")으로 구현하고, 문서 폭증 문제를 RAG 시스템 연동으로 해결한 뒤, 최종적으로 "사람이 할 일 vs 에이전트에 위임할 일"의 거버넌스 설계에 도달한 실전기를 공유한다.

주요 논점:
- **문제의식**: 좋은 AI 답변이 채팅 세션에 갇혀 휘발, 회의록·개인 메모 분산 → 사람도 AI도 매번 맥락 재설명 비용 지불
- **LLM Wiki vs RAG**: RAG = 원본을 청크로 "일단 때려넣고" 질문 시점(query time)에 재검색·재조립. LLM Wiki = 문서 유입 시점(write time)에 요약·결론·링크를 "컴파일"해 다음 질문의 기본 컨텍스트로 사용
- **Karpathy의 3계층**: ① Raw 원본(불변의 source of truth) ② 위키 레이어(요약·개념·최종 결정·문서 간 연결 = LLM이 유지하는 지식 그래프) ③ 스키마/규칙 레이어(읽기·쓰기·저장 규칙으로 "쓰레기 산" 방지)
- **팀 구현 (Knowledge Context)**: Git repo 하나에 `context/`(팀 공유)·`members/<이름>/`(개인)·`agents/`(양쪽에 존재)·`_private`(비동기화) 구조. 비개발자는 Obsidian 뷰, 개발자는 Git 뷰(PR·리뷰·롤백). `AGENTS.md`가 공식 시스템 프롬프트
- **지식 승격 프로세스**: 개인 폴더 초안 + 원본 보관 → 에이전트가 매일 PR 작성 → 자동 검증 + 에이전트 리뷰(규칙·기존 지식 충돌·배치 추천) → **사람이 최종 머지 승인** → `context/` 팀 지식 확정. 비개발자는 "이거 팀에 공유해줘" 한마디로 전 과정 수행
- **문서 폭증 문제와 RAG 연동**: 에이전트가 문서를 무한 생성 → 인덱스 파일 1,000~3,000줄짜리가 20~30개로 관리 불능 → 동기화 워커를 두고 문서 RAG 시스템(자막상 "오픈 바이킹"⚠️) 연동. 파일시스템 패러다임, L0→L2 계층적 로딩(정제 데이터 우선), 재귀 디렉터리 검색(한글 검색 한계 보완), 중복 문서 병합 제안 등 자기진화
- **거버넌스 결론**: "사람이 할 일을 먼저 정의하고 나머지를 위임" — `_private` 관리·팀 지식 삭제·민감 권한/보안은 사람, 그 외는 하네스 안에서 에이전트

> ⚠️ **고유명사·수치 주의**: 본 영상은 발표자 개인·소속 스타트업의 구현 경험담으로, 효과 서술은 정성적 평가입니다. 도구명 다수가 자동자막 기준이라 원명이 불확실합니다("오픈 바이킹"=문서 RAG 시스템, "오픈클로"·"헤르메스 에이전트"=Slack 연동 에이전트, 원 표기 미확인). 인덱스 1,000~3,000줄·파일 20~30개 등 수치는 발표자 발언 기준입니다. 도입 검토 시 해당 도구의 공식 문서로 원명·기능 확인 필요.

## 주제별 정리

### 1. 문제의식 — 흩어진 지식의 반복 비용
- "지난주에 분명히 얘기했는데" — 사람 간 회의에서도, Claude Code/Codex에 컨텍스트를 넣을 때도 같은 설명을 반복
- 마음에 드는 AI 답변이 채팅 세션 안에 갇혀, 따로 복사해두지 않으면 휘발
- 회의록은 막내가 수기 정리(결정과 근거가 분리), 개인 메모는 메신저로 공유하지 않으면 팀이 모르는 지식으로 잔존
- 지식이 흩어져 있으면 사람도 AI도 매번 검색·맥락 이해 비용을 지불

### 2. LLM Wiki 개념 (Andrej Karpathy) vs RAG
- **RAG**: 원본 문서를 청크로 분할 보관, 질문 시점에 관련 청크를 재검색·재조립(query-time retrieval). "뭐가 들어왔는지 모르지만 일단 때려넣는" 저장소 — 설계는 쉽지만 문서 품질 보존이 어려워 "쓰레기 산" 위험
- **LLM Wiki**: 문서가 처음 들어올 때 다른 문서·인덱스와 연결되어야 함. 원본을 그대로 두지 않고 에이전트의 방식으로 한 번 정리·결론을 내린 뒤, 그 정제본을 다음 질문의 기본 컨텍스트로 사용 — "write-time compile"
- 별칭: 세컨드 브레인, (발표자 사내 명칭) Knowledge Context
- **Karpathy 3계층**: ① Raw data source — 불변의 source of truth, 특정 경로 보관 ② 위키 레이어 — 요약·개념·최종 결정·문서 간 연결(LLM이 유지하는 지식 그래프) ③ 스키마/규칙 — 읽기·쓰기·저장 규칙(과다 적재 방지, 즉시 검색)
- Karpathy의 최소 구현: `index.md`(전 문서 인덱스, 원본 연결) + `log.md`(에이전트와의 문서 추가·삭제를 시간순 기록) — 개인용으로는 충분하나 회사·팀에는 부적합
- **가치의 핵심**: 초기 문서 구조 설계가 아니라 **유지보수(품질 보존)의 자동화**

### 3. 팀 구현 — Git + Obsidian "Knowledge Context"
- 저장소 구조 (전체가 하나의 Git repo):
  - `context/` — 팀 합의된 공유 레이어 (회사·팀 차원 공유 지식)
  - `members/<이름>/` — 개인 업무 공간 (개발 프로젝트·문서·기획·디자인)
  - `agents/` — **양쪽에 모두 존재**: 회사 공유 에이전트 지식(스킬·플러그인)과 개인 에이전트 지식. 개인이 만든 스킬을 정식 절차로 팀 공유로 승격
  - `_private` — 민감·비공유 자료, Git 동기화 제외 규칙
- **두 가지 뷰**: 비개발자는 Obsidian(vault)으로 노트 작성, 개발자는 Git으로 repo 전체 관리(PR, 코드오너, 리뷰, 롤백)
- **`AGENTS.md` = 공식 시스템 프롬프트**: 에이전트가 저장소를 임의로 다루지 않고 정해진 규칙 안에서만 작업

### 4. 지식 승격(Promotion) 프로세스
1. `members/<이름>/`에 초안 작성, 근거 원본은 `source/` 경로에 함께 보관
2. 에이전트가 (매일 1회) 팀 지식 승격 제안 PR 작성
3. 자동 검증 + 에이전트 리뷰: 규칙 준수 여부, 기존 지식과의 충돌, 적정 배치 위치 추천
4. **사람이 최종 머지 승인** → `context/`의 팀 지식으로 확정
- 비개발자 배려: PR 개념을 몰라도 "이거 팀에 공유하고 싶어" 한마디로 에이전트가 전 리뷰 프로세스 수행
- Git 기반이므로 누가·언제·무엇을 변경했는지 추적·통제·검색 가능
- **사람 승인 생략 시 즉시 쓰레기장화** — 공식 지식 vs 개인 생각의 최종 판단은 반드시 사람

### 5. 도입 효과
- 컨텍스트 재설명 소멸: Slack에 문서 링크 하나로 대체
- 모두가 토큰(컨텍스트) 절약을 위해 자발적으로 문서 정제, 스킬 제작·공유 문화 확산
- Google Docs 대비 신뢰도 향상: "final_last_final" 혼란을 원본 레퍼런스 역추적으로 해소
- 팀 지식과 개인 지식의 융합 → 형식지를 넘어 **암묵지(tacit knowledge)**를 담아내는 기반 가능성

### 6. 문서 폭증 문제와 RAG 연동
- 에이전트가 문서를 무한 생성 → 인덱스 md 파일이 1,000~1,500줄(3,000줄 미만으로 억지 분절)짜리 20~30개로 증가, 관리 불능
- 해결: 동기화 워커를 중간에 두고 **규칙화된 문서 저장소를 통째로 RAG에 메타데이터 정리·적재** — "규칙화된 저장소는 오히려 RAG하기 쉬운 구성"
- 에이전트의 다회 검색은 이제 어렵지 않음: 질문→검색→컨텍스트 결합→부족하면 재검색 반복
- 채택 시스템(자막상 "오픈 바이킹"⚠️, 원명 미확인)의 선택 이유:
  - 파일시스템 패러다임: 메모리·리소스·스킬을 단일 경로 체계로 통합 → Git repo와 자연 연동
  - L0→L2 계층적 데이터 로딩: 원본이 아닌 정제 데이터부터 컨텍스트 투입
  - 재귀 디렉터리 검색: 한글 검색 한계를 디렉터리 탐색으로 보완
  - 관리 에이전트 하나로 옵저버빌리티 확보 + 자기진화(중복 문서 발견·병합 제안 → 암묵지 발굴)
- 벡터 인덱스는 "검색 상태"일 뿐 — 인덱스 지속 관리 프로세스와 검색 프로세스를 분리 운영
- 연동 후: Slack에 10여 개 에이전트 상주, 업무 대화·회의록 자동 저장이 문서 저장소로 직결(자막상 "오픈클로"·"헤르메스 에이전트"⚠️)

### 7. 거버넌스 — 사람과 에이전트의 역할 구분
- 시스템 도입으로 끝나지 않음 → "AI에게 어디까지 맡길 것인가"
- 전략: **사람이 할 일을 먼저 정의하고, 귀찮은 나머지를 에이전트에 위임**
  - 사람 전담: `_private` 관리, 팀 지식의 **삭제** 결정(Git 히스토리에 기록·검색 가능), 민감 권한·보안
  - 에이전트: 그 외 전부 (하네스 규칙 안에서)
- Obsidian Templater 확장의 frontmatter를 확장해 에이전트 관리 문서 메타데이터 구성, 전부 Git 업로드로 변경점 공개

### 8. 도구 팁
- Obsidian은 CLI 도구 보유 → Claude Code/Codex 연동 용이
- 발표 이틀 전 Obsidian CEO가 "Obsidian 스킬" 공개(발표자 언급⚠️)
- 결론: Knowledge Context = LLM Wiki 아이디어 + Git 동기화 시스템 + Obsidian 운영
